# S3 / R2 — evidence storage

## What it powers

Where signed evidence PDFs are stored. The default is local disk (dev/CI);
in production, an S3-compatible bucket — AWS S3 or Cloudflare R2 — ideally
with **Object Lock** so evidence is immutable for its retention window.
Stored objects are ciphertext (the evidence Service encrypts before storing).

## Code status — complete (build + signing verified)

- `internal/evidence` — `S3Store` implements the storage interface against
  the S3 REST API, signing every request with hand-rolled **AWS Signature
  Version 4** (`sigv4.go`), no SDK.
- The SigV4 signer is unit-tested against AWS's published `get-vanilla`
  test vector. The store itself is build-verified; it has not been run
  against a live bucket from this environment.
- With `EVIDENCE_S3_BUCKET` unset the app uses `LocalStore` (disk).

## Setup

### Option A — AWS S3

1. **Create a bucket with Object Lock enabled** — Object Lock can only be
   turned on *at bucket creation*.
2. Set a **default retention**: mode `COMPLIANCE`, ~7 years (matches the
   app's evidence retention). Every uploaded object then inherits it and is
   immutable until it expires — no per-upload setting needed.
3. Create an IAM user with `s3:PutObject`, `s3:GetObject`, `s3:DeleteObject`
   on `arn:aws:s3:::YOUR_BUCKET/*`; generate its access key + secret.
4. Leave `EVIDENCE_S3_ENDPOINT` empty — the regional endpoint is derived
   from `EVIDENCE_S3_REGION`.

### Option B — Cloudflare R2

1. Create an R2 bucket (enable Object Lock on it if you want immutability).
2. Create an **R2 API token** with object read/write — it yields an access
   key ID and secret.
3. `EVIDENCE_S3_ENDPOINT` = `https://<ACCOUNT_ID>.r2.cloudflarestorage.com`,
   `EVIDENCE_S3_REGION` = `auto`.

### Environment variables

A bucket value switches the store from local disk to S3.

| Variable | Required | Value |
|---|---|---|
| `EVIDENCE_S3_BUCKET` | yes | Bucket name |
| `EVIDENCE_S3_REGION` | yes | AWS region, or `auto` for R2 |
| `EVIDENCE_S3_ENDPOINT` | R2/MinIO only | Account/cluster endpoint URL |
| `EVIDENCE_S3_ACCESS_KEY_ID` | yes | Access key ID |
| `EVIDENCE_S3_SECRET_ACCESS_KEY` | yes | Secret access key |

## Verify

1. Restart; the log should read `evidence store: s3-compatible bucket`.
2. Run a drill; confirm a `<drill-id>.pdf` object appears in the bucket.
3. Download the evidence from the drill page — it should decrypt and open.

## Object Lock and deletion

With a COMPLIANCE retention, the bucket itself refuses to delete an object
before its lock expires — that is the immutability guarantee. Two
consequences, both intentional:

- The retention sweeper's delete only succeeds once `retain_until` has
  passed; that is exactly the policy.
- A GDPR hard delete cannot remove a still-locked object — but crypto-shred
  destroys the account's encryption key, so the locked ciphertext is
  permanently undecryptable regardless.
