# RFC 3161 timestamp authority

## Status — seam (documented, not implemented)

Like the document-signing cert, this is deliberately **not** coded: a
trusted timestamp is only worth anything if its ASN.1 token is parsed and
verified correctly, and a hand-rolled, unverified implementation for a
legal-evidence feature would undermine the claim it is meant to support.
This runbook documents the seam, what a real implementation needs, and why
the current approach is a sound interim.

## What it powers

*Proof of when* — independent, third-party attestation that a drill's
evidence existed at a particular point in time, rather than relying on the
app's own word for it.

## Current state — self-asserted `signed_at` (works today)

`internal/evidence` records a `signed_at` timestamp on every signature and
**binds it into the Ed25519 signature**: the signed message is
`sha256(pdf) | signed_at` (`signedMessage` in `sign.go`). So the timestamp
is genuinely tamper-evident — it cannot be altered after the fact without
breaking the signature.

Its limitation: `signed_at` is the **app's own clock**. The signature
proves the evidence and the timestamp were signed *together*, but nothing
stops the signer from putting any time it likes into `signed_at`. A
relying party has to trust Restore Drill not to backdate or postdate.

## What an RFC 3161 timestamp adds

RFC 3161 defines a Time-Stamp Authority (TSA): an independent service that,
given a hash, returns a signed token attesting that hash was presented to
it at a specific time. The time comes from the TSA's clock, not ours, and
is signed by the TSA's own certificate. It converts "we say it was signed
at T" into "an independent TSA certifies this hash existed by T" —
verifiable by anyone, with no trust in Restore Drill required.

## Why it is a seam, not built

1. **ASN.1, not JSON.** RFC 3161 is DER-encoded ASN.1: the request is a
   `TimeStampReq`, the response a `TimeStampResp` wrapping a CMS
   `SignedData` token. Producing and *verifying* that correctly (hash
   match, TSA cert chain, signed-time extraction) is exactly the kind of
   fiddly crypto-adjacent parsing that must not ship unverified.
2. **Pairs with the signing cert.** A TSA token's trust chain is verified
   the same way as the document-signing cert's (CMS + a CA root). The two
   seams share that machinery — they are naturally built together, against
   real test vectors, not piecemeal.
3. Shipping unverified ASN.1 for a legal-evidence feature is the precise
   risk this runbook avoids.

## How to implement it (bounded follow-up)

1. Pick an RFC 3161 TSA (e.g. DigiCert, Sectigo, or a free TSA such as
   FreeTSA) and note its endpoint URL.
2. Add a TSA client: build a `TimeStampReq` over `sha256(pdf)`, POST it to
   the TSA endpoint (`Content-Type: application/timestamp-query`), and
   parse the `TimeStampResp`.
3. After `Signer.Sign`, request a token and store the returned timestamp
   token blob on the signature row (a new `evidence_signatures` column,
   e.g. `tsa_token`).
4. `Verify` additionally validates the token: the hash inside matches
   `pdf_sha256`, the TSA cert chains to a trusted root, and the genTime is
   surfaced as the authoritative timestamp.
5. Carry the TSA cert serial/fingerprint alongside the token so a TSA key
   rotation does not invalidate older evidence.

Until then, the Ed25519-bound `signed_at` is a sound interim: the
timestamp is tamper-evident and bound to the evidence — it simply rests on
Restore Drill's clock rather than an independent authority's.
